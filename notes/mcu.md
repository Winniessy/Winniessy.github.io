# MCU的中断

Date: 2026-07-13  

一、轮询和中断的区别、场景，为什么这么用？
一句话： 轮询适用于需要周期性去捕获，并且*处理速度比较快，实时性要求比较低的（迟几微秒都ok）的事件；而中断是异步触发，事件发生频率比较小，并且需要实时响应。
轮询是“忙等”，适合简单、高速、确定性的场景；中断是“异步通知”，适合复杂、低功耗、多并发的场景。
轮询是CPU主动、周期性地去查询设备状态，是‘同步’的；中断是设备主动通知CPU，是‘异步’的。
二、中断需要注意些什么？
一、中断服务函数必须“短小精悍”
原则：中断里只做最紧急、最少量的事情，复杂逻辑放到主循环或任务中处理。
- 避免延时/阻塞：绝对不能在ISR中调用 delay()、while 等待标志位等阻塞函数。中断的本质是抢占，长时间占用会阻塞其他中断或主程序，导致系统响应变慢甚至看门狗复位。
- 避免复杂运算：浮点运算、大数组操作、复杂算法都应当避免，它们会显著增加中断响应时间。
- 避免非重入函数：printf()、malloc() 等通常不是线程安全的，且可能涉及系统调用或锁，在中断中使用极易死锁。
- 典型做法：ISR中仅做关键操作——清除中断标志、读取硬件寄存器、将数据放入缓冲区（如循环队列）、释放信号量或发送消息给任务，然后立即返回。
1. 使用 volatile 修饰共享变量
- 作用：告诉编译器，该变量的值可能被硬件或中断随时改变，禁止优化（如缓存到寄存器、重排序）。
- 注意：volatile 只能保证每次访问都从内存地址读取，不能保证原子性。对于8位或16位变量的单次读写，在多数架构上是原子的；但对于32位变量在8位机上、结构体、多语句操作（如 count++），仍需要临界区保护。
2. 临界区保护——关中断
- 原理：在主程序访问共享资源时，暂时关闭全局中断（或仅关该中断），防止ISR中途打断。
- 实现：
- c
- // 常用宏（如STM32的HAL）__disable_irq();// 访问共享资源__enable_irq();
- 风险：关中断时间过长会导致中断延迟增加，可能丢失实时事件。必须保证临界区极短，只做必要的读写，绝不能在关中断时做耗时操作。
- 中断嵌套情况：如果只关闭当前优先级的中断，更高优先级的中断仍可能打断，需要根据芯片特性选择合适的关中断方式（如ARM Cortex-M 的 __set_PRIMASK(1) 关闭所有可屏蔽中断）。
3. 互斥锁/信号量（适用于RTOS）
- 在有操作系统的环境中，ISR 与任务之间共享数据通常用二进制信号量或消息队列。ISR 中释放信号量，任务中获取信号量后再处理数据。
- 这种方式的优点是临界区极短，且不关中断（仅通过调度器锁定），但对裸机程序不适用。
4. 原子操作与专用指令
现代MCU（如ARM Cortex-M）提供原子操作指令（LDREX/STREX）或位带操作，可以用来实现无锁的数据访问。对于简单的增减操作，可以使用原子操作函数（如 __sync_fetch_and_add 或CMSIS中的 __atomic_*）。
三、区分中断优先级分组与嵌套
中断嵌套是导致栈溢出、数据错乱的重要原因。
- 优先级分组：ARM Cortex-M 等内核支持中断优先级分组，分为抢占优先级和子优先级。只有抢占优先级高的中断才能打断正在执行的ISR。合理分配优先级，确保关键中断（如电机保护、通信错误）有最高抢占优先级，普通中断（如按键）可以设低。
- 避免嵌套过多：嵌套层数过深会消耗大量栈空间，可能导致栈溢出。在资源紧张的单片机中，可以设计所有中断均为同一抢占优先级（即不支持嵌套），简化栈空间管理。
- 注意“优先级反转”：如果低优先级中断持有了某个资源，高优先级中断又需要该资源，会导致高优先级中断长时间等待。解决方法：尽量减少资源占用时间，或使用优先级继承机制（RTOS中）。
四、中断嵌套的问题：
一、中断嵌套与栈溢出的关系
每次中断发生时，CPU会自动将以下内容压入当前使用的栈（主栈MSP或进程栈PSP）：
1. 8个寄存器（xPSR, PC, LR, R12, R3~R0）—— 固定 8 × 4 = 32 字节（Cortex-M架构）
2. 如果使用了浮点单元（STM32F411有FPU，F103没有），还会额外压入浮点寄存器，约增加 34 × 4 = 136 字节（但可配置是否压栈）
3. ISR内部：局部变量、函数调用时的LR压栈、可能的嵌套函数调用
因此，每一次中断嵌套，栈消耗 = 固定硬件压栈 + 软件动态消耗。

二、如何估算最大安全嵌套深度
5. 确定栈总大小
- 裸机程序：使用主栈（MSP），大小在启动文件（如 startup_stm32fxxx.s）中定义，常见值：Stack_Size EQU 0x400（1KB）或 0x1000（4KB）
- RTOS：每个任务有自己的栈，中断使用主栈（或单独中断栈），大小通常由 FreeRTOSConfig.h 中 configISR_STACK_SIZE 或启动文件中的主栈决定
2. 计算单次中断的栈消耗
- 硬件压栈：32字节（无FPU）/ 32+136=168字节（有FPU且开启自动压栈）
- ISR代码：如果ISR仅清标志、设置标志、释放信号量，局部变量极少，软件消耗通常 < 32 字节
- 函数调用：如果ISR内调用其他函数，每次调用额外压入LR（4字节）和可能的局部变量
3. 估算最大嵌套层数
假设：
- 主栈大小 = 1KB (1024字节)
- 无FPU，硬件压栈固定32字节
- ISR软件消耗约 32 字节（简单ISR）
- 每次中断嵌套总消耗 ≈ 64 字节
则理论最大嵌套深度 = 1024 ÷ 64 ≈ 16 层
但如果：
- 栈只有 256 字节 → 只能嵌套 4 层
- 栈有 4KB → 可以嵌套 64 层（但实际嵌套受优先级级数限制，最多16级）

---
三、实际中嵌套深度受两个上限制约
1. 优先级级数上限
Cortex-M的优先级分组最多提供16个抢占优先级，所以理论上最多嵌套16层（从最高优先级到最低优先级逐层打断）。
即使栈空间足够，也不会超过16层（除非有相同优先级的中断不能嵌套）。
2. 栈空间上限
实际栈大小会限制更小的嵌套深度。例如，如果栈只有 512 字节，而每次中断消耗 100 字节，则最多嵌套 5 层。
因此，安全嵌套深度 = min(优先级级数, 栈空间允许的深度)。

五、如何防止栈溢出导致的中断嵌套崩溃
1. 合理配置栈大小
在启动文件中根据系统需求配置 Stack_Size，建议至少 1KB 起步，复杂系统 4KB 或更大。
2. ISR 保持极简
严格遵循“短小精悍”原则，避免在 ISR 内调用函数（除非必要），避免局部大数组。
3. 使用栈检测
  - 在启动时用固定模式填充栈区（如 0xAA），运行后检查栈使用情况
  - 使用调试工具（如 STM32CubeMonitor）观察栈余量
  - RTOS 中开启栈溢出检测功能（如 FreeRTOS 的 configCHECK_FOR_STACK_OVERFLOW）
4. 优先级分组合理
不是所有中断都需要不同的抢占优先级。将不重要的中断设为同级（不嵌套），减少嵌套深度风险。
5. 裸机下使用 MSP 主栈
确保中断和主循环共用栈时，主循环不会过度消耗栈空间。可以估算主循环的最大栈使用（局部变量、函数调用），加上最大中断嵌套消耗，总和应小于栈大小。

六、总结回答面试官
“中断嵌套的最大深度没有固定数值，它受两个因素制约：一是 NVIC 优先级配置决定的抢占优先级级数（最多16级）；二是实际可用的栈空间大小。每次中断嵌套会消耗固定硬件压栈（32字节，若启用FPU则更多）加上 ISR 内部的局部变量和函数调用开销。如果栈空间足够大（如4KB），且 ISR 设计极简，理论上可以达到16层嵌套；但如果栈只有512字节，且 ISR 较复杂，可能只能安全嵌套3~5层。工程上我会通过合理配置栈大小、精简 ISR 代码、使用栈检测工具来确保不发生溢出。”
