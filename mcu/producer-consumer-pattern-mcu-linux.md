# 生产者—消费者模式：MCU 与 Linux 中的典型应用

生产者—消费者模式用于解耦“数据产生”和“数据处理”：

```text
生产者 → 缓冲区 / 队列 → 消费者
```

它适合生产速度和消费速度不一致，或者两者运行在不同执行上下文中的场景。缓冲区只能吸收短时间的速率波动；如果生产速度长期高于消费速度，队列最终仍会满。

## 一、四种基本模型

| 模型   | 含义        | 常见场景            |
| ---- | --------- | --------------- |
| SPSC | 单生产者、单消费者 | UART ISR → 解析任务 |
| MPSC | 多生产者、单消费者 | 多任务 → 日志任务      |
| SPMC | 单生产者、多消费者 | 请求队列 → Worker 池 |
| MPMC | 多生产者、多消费者 | 多核线程池、网络服务器     |

SPSC 最简单，通常只需要生产者维护写指针、消费者维护读指针，适合环形缓冲区。

MPSC、SPMC 和 MPMC 存在多个执行上下文竞争同一读写位置的问题，通常需要锁、临界区、原子操作或操作系统提供的线程安全队列。

## 二、竞争消费与发布—订阅

多个消费者不代表每个消费者都能收到全部数据。

普通工作队列采用竞争消费：

```text
生产数据：A B C D

消费者 1：A C
消费者 2：B D
```

每条数据只由一个消费者处理。

如果显示、存储和网络任务都需要收到同一份数据，则应使用：

* 发布—订阅；
* 每个消费者独立队列；
* 多读指针缓冲区；
* 引用计数共享数据。

## 三、环形缓冲区的作用

环形缓冲区只是生产者—消费者模式的一种实现。

它适合嵌入式系统，因为具有以下特点：

* 固定内存；
* 不需要频繁动态分配；
* 读写通常为 O(1)；
* 适合连续字节流；
* 适合 ISR、DMA 与任务之间传递数据。

典型结构：

```c
typedef struct {
    uint8_t buffer[BUFFER_SIZE];
    volatile uint32_t write_index;
    volatile uint32_t read_index;
} RingBuffer;
```

需要注意，`volatile` 不能自动保证线程安全。在多核系统中，还可能需要原子操作和内存屏障。

## 四、MCU 中的典型场景

### 1. ISR 生产，任务消费

```text
UART ISR → 环形缓冲区 → 协议解析任务
CAN ISR  → 消息队列   → CAN 处理任务
GPIO ISR → 事件队列   → 按键任务
```

中断中只完成快速的数据接收和通知，复杂处理放到任务上下文中。

### 2. DMA 采集，算法任务处理

```text
ADC / 音频 DMA → 数据块队列 → 滤波、FFT、编码任务
```

常见方式是双缓冲：

```text
DMA 填充 Buffer A
CPU 处理 Buffer B
```

然后双方交换缓冲区。

### 3. 多任务统一日志

```text
网络任务 ─┐
控制任务 ─┼→ 日志队列 → 日志任务 → UART / Flash
传感器任务 ┘
```

这是典型的 MPSC，可以避免多个任务直接竞争串口或文件系统。

### 4. 设备服务任务

```text
配置任务 ─┐
日志任务 ─┼→ Flash 请求队列 → Flash 服务任务
升级任务 ─┘
```

只有一个任务真正访问设备，从而简化设备状态、DMA 和错误处理。

## 五、FreeRTOS 中的常用实现

* **Queue**：适合固定大小消息，支持多生产者和多消费者。
* **Stream Buffer**：适合连续字节流，主要面向 SPSC。
* **Message Buffer**：适合变长消息，并保留消息边界。
* **环形缓冲区 + 信号量**：缓冲区保存数据，信号量负责通知任务。

例如 ISR 向队列发送数据：

```c
BaseType_t task_woken = pdFALSE;

xQueueSendFromISR(queue, &message, &task_woken);
portYIELD_FROM_ISR(task_woken);
```

如果唤醒了更高优先级任务，退出中断时可以立即触发任务切换。

## 六、Linux 中的典型场景

### 1. 网络服务器线程池

```text
监听线程 → 请求队列 → 多个 Worker
```

如果有多个 I/O 线程同时提交请求，则属于 MPMC。

### 2. 音视频和 AI 流水线

```text
采集 → 解码 → 图像处理 → AI 推理 → 编码
```

每一级既是上一阶段的消费者，也是下一阶段的生产者。

### 3. 驱动与内核缓冲区

```text
硬件 / DMA → 内核 Ring Buffer → 用户进程
```

常见于：

* 网卡 RX/TX Ring；
* ALSA 音频；
* V4L2 视频缓冲区；
* `perf`、`ftrace`；
* eBPF Ring Buffer；
* `io_uring`。

### 4. 中断延迟处理

```text
硬中断 → workqueue / threaded IRQ → 后续处理
```

其思想与 MCU 中的：

```text
ISR → RTOS Queue → 任务
```

基本一致。

## 七、MCU 与 Linux 的主要区别

| 对比项    | MCU                | Linux                                  |
| ------ | ------------------ | -------------------------------------- |
| 主要目标   | 实时性、确定性            | 吞吐量、扩展性                                |
| 并发形式   | 单核任务切换、ISR 抢占      | 多核真正并行                                 |
| 内存     | 固定、静态分配            | 动态内存、虚拟内存                              |
| 常见生产者  | ISR、DMA、任务         | 线程、进程、驱动、硬件                            |
| 主要同步方式 | 临界区、信号量、RTOS Queue | mutex、condition variable、futex、eventfd |
| 额外问题   | 中断延迟、队列容量          | Cache、一致性、内存屏障、false sharing           |

MCU 即使是单核，也存在任务与 ISR 之间的并发问题。

Linux 多核环境中，不同 CPU 可以同时访问队列，因此更需要关注：

* 原子操作；
* 内存顺序；
* Cache 一致性；
* 自旋锁；
* false sharing。

## 八、队列空和队列满

消费者遇到空队列时，可以：

* 忙轮询；
* 非阻塞返回；
* 阻塞等待；
* 等待信号量或事件通知。

生产者遇到满队列时，可以：

* 丢弃新数据；
* 覆盖旧数据；
* 阻塞等待；
* 返回错误；
* 启用流控或背压。

具体策略取决于数据语义：

* 日志可以允许丢弃；
* 最新状态可以覆盖旧数据；
* 控制命令通常不能随意丢失；
* ISR 通常不能阻塞等待。

## 九、如何选择

* UART、音频等连续数据：环形缓冲区或 Stream Buffer。
* CAN 帧、控制命令等固定消息：Queue。
* 变长协议消息：Message Buffer 或消息头加字节流。
* 图像、音频块等大数据：传递“地址 + 长度”，避免复制。
* 多消费者都需要完整数据：发布—订阅。
* 多个消费者只需分担工作：竞争式工作队列。

## 十、总结

生产者—消费者模式是一种并发设计思想，并不等同于环形缓冲区。

它的核心作用是：

* 解耦数据产生和处理；
* 吸收短时间的速率差异；
* 将中断或 I/O 快速路径与复杂处理分离；
* 通过队列控制并发和数据所有权。

MCU 更关注实时性、固定内存和中断响应；Linux 更关注多核并行、吞吐量和进程隔离。设计时必须明确生产者数量、消费者数量、消息语义、队列满策略以及数据所有权。
